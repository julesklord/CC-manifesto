v1.1 — April 2026 Philosophy for responsible development with AI
01 — The Problem

We stopped thinking.

The software industry replaced mechanical friction with infinite AI generation and called it progress. The bottleneck was never typing — it was decision-making. We've outsourced both.

We became syntax accumulators, not craftspeople. In our obsession with velocity, personal branding, and AI-assisted over-engineering, we've forgotten the fundamental purpose: build useful tools that solve real problems without wasting energy — neither ours, nor the system's, nor the planet's.

Twenty npm packages to render a checkbox.

An agent refactoring code that already works because the prompt implied it should.

A README generated by AI describing a project that doesn't exist yet.

Architecture for a million users serving forty.

Code the author cannot debug without reopening the chat that wrote it.

The problem isn't the tools. It's abdication dressed as productivity.

We reclaim control. We refuse to be spectators of the noise. We choose to think first. We use AI as an instrument — never as a substitute for judgment.

02 — Core Principles

Eight commitments.
No exceptions.

1
Principle 01

Logic dictates. AI executes.

Delegating typing is not delegating thinking. The architecture, data flow, and final decision live in the creator's mind. We are directors; AI is the instrumentalist. Every critical decision is auditable to a human origin.

  • The project's mental map exists in your head, not in an AI chat
  • You can explain why each component exists without consulting the history
  • Decisions delegated to AI are documented and justified
  • Code is readable without the original prompt
2
Principle 02

Sufficiency is victory.

Every system has a natural limit. Like an audio mix, code suffocates under unnecessary layers. When it works and it's stable: stop. Perfectionism is a luxury we cannot afford — and a debt we will pay later.

  • If the function is fulfilled and maintainable, iteration stops
  • You don't anticipate optimizations nobody asked for
  • You don't refactor "just in case" for a future that may not arrive
  • The metric is: works + is understandable — not perfect
3
Principle 03

Conscious decisions, deliberate execution.

Before writing: we think. Clear architecture, explicit dependencies, documented trade-offs. Speed never replaces thought. A slow start saves ten fast restarts.

  • Before touching code: write the problem in 3–4 sentences
  • Critical decisions are recorded (DECISIONS.md)
  • Every external dependency has a documented reason
  • Trade-offs are explicit in comments, not assumed
4
Principle 04

Responsible energy spending.

Deploying massive compute to refactor what already works is waste. We privilege "good enough" over sterile perfection. Every AI token has a cost — financial, cognitive, and environmental. We use it where it genuinely matters.

  • You don't run massive models for trivial use cases
  • If the manual solution is cheaper and faster, do it manually
  • You document why AI was used in each critical case
  • You don't generate code "just in case you need it later"
  • Your cognitive energy is also a finite resource — protect it
5
Principle 05

Evidence drives iteration.

We don't chase phantom problems. We deploy early, observe honestly, iterate only with evidence. Speculation is not a reason to refactor. Real errors from real usage are.

  • Code goes to production early, with monitoring
  • First changes come from real errors, not gut feelings
  • Metrics that matter: does it work? is it used? where does it fail?
  • 5+ users reporting the same issue means fix it — 1 person speculating means ignore it
  • Refactoring requires a specific trigger: a condition, not a schedule
6
Principle 06

Purpose over exhibition.

We build tools that solve our real workflow — not tools that demonstrate our capability to an imagined audience. The only valid measure: does this make the work easier? GitHub stars, trending repos, and social visibility are irrelevant signals.

  • If the code is never shown publicly, all the better
  • Success isn't recognition — it's a workflow that no longer hurts
  • You document for yourself in six months, not for an audience
  • Tools evolve with real needs, not with trends
7
Principle 07 v1.1

Cognitive sovereignty is non-negotiable.

AI can generate your code. It cannot generate your understanding. If you cannot debug, rewrite, or explain the core of your project without AI assistance, you are not the author — you are the operator. The difference matters when something fails in production at 2am with no connectivity.

There is a critical difference between "I can read this" and "I understand this." The real test: could you rebuild the critical module from scratch, given only the spec? If not — that's cognitive debt, not just technical debt. AI should accelerate your existing skills, never replace the need to have them.

  • Once per project: attempt to rewrite the core module from scratch in 30 minutes
  • If you can't: either learn it first, or document the section as a black box (with explicit risk)
  • The real test: can you debug it without AI context?
  • Calibrate trust by domain — AI is reliable for boilerplate, dangerous for niche internals
  • If you catch AI hallucinating in your domain: log it as calibration data
8
Principle 08 v1.1

Failures are primary documentation.

An annotated error log is worth more than any up-front design document. Failure modes discovered in production are irreplaceable knowledge. We document failures with the same rigor as successes — because they carry more signal.

This applies doubly to AI failures: hallucinated APIs, wrong assumptions, subtly broken patterns. Each one is a data point for calibrating trust. Ignoring AI failures is ignoring the most honest feedback in your workflow.

  • FAILURES.md carries the same institutional weight as DECISIONS.md
  • Every resolved bug: what assumption failed? That goes in the doc.
  • AI errors (hallucinations, incorrect code) are logged as domain calibration data
  • Failure documentation is not blame — it's institutional memory
  • The question after every incident: what did this teach us that we didn't know before?
03 — How This Translates to Action

Before, during, and after.

Before Coding

Define the Problem — not the solution
  • What fails now? Where exactly?
  • Who uses it? How often?
  • What's the cost of NOT fixing it?
Set hard complexity limits
  • Maximum external dependencies? (write the number)
  • Maximum lines before mandatory refactor? (write the number)
  • What cases will NOT be supported? (be explicit and honest)
Estimate true energy cost
  • If using AI: why not a simpler solution?
  • Is AI use justified versus maintaining old code?
  • What technical debt is being accepted today?
  • What is the cognitive cost to the team?

During Coding

Keep the mental map alive
  • Can you explain each decision without consulting the AI?
  • Are AI-suggested decisions commented and attributed?
  • Is the code readable without the chat history?
Maintain the audit trail
  • If AI suggested something critical: document it immediately
  • Format: [Decision] | [Alternatives rejected] | [Why] | [Who] | [Date]
  • AI hallucinations caught → log in FAILURES.md with domain
Testing = thinking, not vanity coverage
  • Do tests verify expected behavior — or just "pass"?
  • Are there tests for real edge cases, not invented ones?
  • Metric: can you sleep if a test fails? If not, it's important.
Documentation directed at yourself
  • Write for yourself in six months — not for an imagined audience
  • If AI generated it: do you fully understand it? Would you simplify it?
  • If you wouldn't simplify it: document why the complexity is justified

After Deployment

Real observation, not vanity metrics
  • First two weeks: what fails? What goes unused?
  • Real metrics: logs, errors, actual usage — not impressions
Iterate only with evidence
  • 5+ users with the same issue? Fix it.
  • Speculative optimization idea? Ignore it.
  • 80/20 rule applied? Document it.
Make technical debt visible
  • Every HACK: why? What condition triggers refactor?
  • File: TECHNICAL_DEBT.md — priority: critical / medium / low
  • Review quarterly: resolve, escalate, or consciously defer
04 — Exceptions

When to break these rules.

A manifesto without explicit exceptions is a religion, not a methodology. These rules exist for production code and maintained projects. They do not apply when:

Disposable Prototype

You're building a throwaway spike (<48h). Skip the docs. If it survives 48h, apply the full standard retroactively.

Active Learning

You're exploring a new API, framework, or domain. Mess freely. The constraint is: document what you learned, not what you built.

Emergency Incident

Production is down. Fix first, document after. Every exception taken during an incident gets a post-mortem entry.

Explicit Override

You knowingly break a principle. Fine — but document the override, the reason, and the exit condition. Conscious deviation is not a violation.

The judgment call is always yours. These are defaults, not constraints. The standard only fails when you break rules unconsciously.

06 — The Standard

If the checklist is complete.

This is not a public certification. It's an internal standard — the FMG mark that a project was built with discipline, not speed theater. The badge means the work can be maintained, explained, and handed off by its author. That's it.

✦
Conscious Code — FMG Standard v1.1
  • Logic → Explicit architecture, human origin
  • Sufficiency → No over-engineering, no ghost features
  • Decisions → Fully auditable, documented
  • Energy → Responsible, justified use
  • Evidence → Verified maturity, real usage
  • Purpose → Utility over exhibition
  • Sovereignty → Author can debug and rebuild without AI
  • Failures → Documented as institutional knowledge

This badge has no external authority. Its value is entirely internal — and that makes it harder to fake. If you certified yourself without completing the checklist, you've only deceived yourself. We consider that worse.

07 — Final Notes

This manifesto is not a list of pretty principles. It is a way of working that demands discipline — not laziness dressed as philosophy.

It's not "do the minimum." It's "do the minimum, thoughtfully, documented, and verified against reality."

If you copy these principles without living them, the badge is just marketing. The goal here is something real: a workflow that you understand, can explain, and can defend — with or without internet access, with or without the chat history that built it.

The measure of your mastery is not what you can generate. It's what you can maintain.
